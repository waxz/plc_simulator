# Conveyor Flow Simulator - Python Engine

This directory contains the Python physics engine and visualizer for the Conveyor Flow Simulator. It works in tandem with the React-based frontend editor to simulate physical interactions between parcels, conveyor belts, and sensors.

## Architecture

The Python simulation runs as a standalone process that:
1. Parses a JSON configuration file generated by the React editor (`config_parser.py`).
2. Runs a 2D rigid-body physics simulation using Pymunk (`physics.py`).
3. Renders the simulation in a standalone window using Pygame (`visualizer.py` & `editor_ui.py`).
4. Evaluates sensor triggers and drives kinematics (belts moving parcels).

### File Structure
- `main.py`: Entry point. Initializes the physics engine and the visualizer loop.
- `physics.py`: The core Pymunk simulation. Handles belt kinematics, vertex-based impulse application, parcel collision, and sensor state evaluation.
- `visualizer.py`: The Pygame rendering loop. Draws belts, sensors, parcels, and selection highlights based on the simulation state.
- `editor_ui.py`: The Pygame GUI panel for viewing read-only properties of the selected component.
- `math_utils.py`: Utility functions for geometry (e.g., ray casting for `point_in_polygon`).
- `config_parser.py`: Loads the JSON config and deserializes it into typed Python dataclasses.

## Coordinate Systems Unified

The simulation bridges mathematical 2D space, Pymunk physics space, React/Konva screen space, and Pygame screen space. To ensure components align perfectly across all domains, a **Unified Coordinate System** is used.

### 1. World Coordinate System (Standard Cartesian)
All `x` and `y` coordinates defined in the configuration JSON are treated as **World Coordinates**:
- **Origin `(0, 0)`**: Fixed reference point.
- **X-Axis**: Increases to the **Right**.
- **Y-Axis**: Increases **Upwards** (Standard Cartesian).
- **Angles**: `0` degrees points Right (+X). `90` degrees points Up (+Y).
- **Rotation Direction**: Positive angles rotate **Counter-Clockwise (CCW)**.

### 2. Pygame / Konva Screen Coordinate System
Both the React frontend (Konva) and the Python visualizer (Pygame) render to screens where the Y-axis is inverted:
- **X-Axis**: Increases to the Right.
- **Y-Axis**: Increases **Downwards**.

### 3. Rendering Rotation Alignment
Because the Screen Y-axis is inverted relative to the World Y-axis, rotating an object mathematically can be tricky. However, the system is perfectly unified because of how Pygame and Konva handle rotation:

- **Pygame (`pygame.transform.rotate(surf, angle)`)**: Rotates a surface CCW on the screen. Because the screen Y-axis is Down, a CCW rotation on the screen effectively rotates the +X axis towards the -Y axis (Up). This exactly mirrors the World CCW rotation! Therefore, we apply the raw `rotation` value directly to Pygame surfaces.
- **Konva**: Rotates surfaces CW by default. The React frontend passes `-rotation` to Konva, which reverses it to CCW, making it perfectly align with both World and Pygame.

### 4. Bounding Boxes and Dimensions
- **Linear Belts**: Have `length` (width) and `beltWidth` (height). At `rotation = 0`, they are horizontally aligned along the X-axis.
- **Sensors**: Have `width` and `height`. At `rotation = 0`, if a sensor is defined as `width = 0.1` and `height = 1.0`, it acts as a tall, vertical bounding box aligned along the Y-axis. 
- **Sensor Beams (Laser Banners)**: Laser beams are always distributed along the local Y-axis of the sensor. In the World space, when `rotation = 0`, the local Y-axis points UP `(0, 1)`. When rotated by `angle`, the local Y direction becomes `(-sin(angle), cos(angle))` due to the CCW rotation matrix.
